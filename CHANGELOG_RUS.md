# История изменений

## v0.13.99

- **FIXED:** Проверка прямого подключения к TG (TG direct) теперь использует `--resolve` для обхода DNS и прямого подключения к IP-адресам дата-центров Telegram (149.154.167.220, 149.154.167.51, 91.108.56.190) — требуется ответ от 2 из 3 ДЦ. Ранее разрешение имен происходило через DNS, который мог вернуть IP-адреса CDN/Cloudflare, что давало ложноположительный результат при блокировках регулятора.
- **NEW:** `check_health` A3: проверяет первый резервный SOCKS (tier2_1) на доступность Telegram API — результат отображается как tier2 ✅/❌ в меню Status (Статус) и Tunnel Health (Здоровье туннеля). Скрыто, если резервные прокси не настроены.
- **UX:** Status и Tunnel Health — индикаторы `tunnel` и `SOCKS` объединены в единый индикатор `tunnel SOCKS5`. Зеленый = транспорт ок + SOCKS работает, желтый = транспорт ок, но порт SOCKS недоступен, красный = ошибка транспорта.
- **NEW:** Защита от отключенного `mixed_proxy` в Probe (Проверке) — показывает понятную ошибку вместо результатов N/A, если mixed_proxy выключен.
- **NEW:** Проверка доступности Clash API в Probe — показывает подсказку "Enable YACD" (Включить YACD), если Clash API недоступен.
- **NEW:** Активная секция при запуске инициализируется из первой настроенной секции podkop (а не жестко задана как `main`).
- **FIXED:** Отображение публичного IP показывает `N/A` вместо `?`, когда IP недоступен (двойной NAT / нет связи).
- **NEW:** Кнопка `Test Fallback` (Проверить резерв) добавлена на верхний уровень Bot Settings (Настройки бота) рядом с пунктом Fallback SOCKS.
- **NEW:** Podkop Settings (Настройки Podkop) — кнопка `Test Proxies` (Проверить прокси) добавлена в меню настроек верхнего уровня.
- **UX:** Все кнопки навигации в меню теперь показывают 🏠 — это устраняет путаницу между "вернуться на один экран назад" (← Back) и "перейти в главное меню" (🏠 Menu).

## v0.13.94

- **NEW:** Проверка активного исходящего соединения (`Probe Active Outbound`) — новая функция в разделе "Диагностика". Тестирует текущий активный прокси через `mixed_proxy` SOCKS без переключения селектора или прерывания трафика. Четыре последовательные проверки: (1) Гео — IP выхода, страна, ASN через ipapi.co; (2) Подсказка Google — парсит поле `MgUcDb` с google.com, показывая, как Google видит страну выхода; (3) Доступность сервисов — YouTube, Telegram API, ChatGPT, Discord через прокси, каждый с классификацией HTTP-статуса (ОК / Редирект / Заблокирован / Таймаут); (4) Пропускная способность — скачивание 200 КБ через speed.cloudflare.com с обнаружением троттлинга провайдера и блокировки 16 КБ. Карточка результатов показывает все данные в одном сообщении с контекстными кнопками действий.
- **NEW:** Обнаружение троттлинга/блокировки РКН в тесте пропускной способности. Три классифицируемых состояния: `ok` (≥0.8 Мбит/с), `throttled` (<0.8 Мбит/с, но данные идут), `block16k` (получено ~14–20 КБ, затем соединение оборвалось — известный паттерн РКН по обрезке трафика после первых 16 КБ), `blocked` (данные не получены). Для каждого состояния отображается соответствующий текст предупреждения, объясняющий паттерн.
- **NEW:** Проверка доступности Telegram API через активный исходящий канал. Обнаруживает ситуации, когда прокси (особенно российские VDS) пропускает общий трафик, но блокирует `api.telegram.org` — объясняет, почему этот прокси нельзя использовать как фоллбек (tier2) для бота. В случае блокировки карточка результатов показывает кнопку "Настройки бота" для удаления его из списка фоллбеков.
- **NEW:** Контекстные кнопки действий в результатах проверки. Если обнаружен троттлинг/блокировка в режиме селектора → кнопка "Сменить прокси" (открывает Исходящие). Если троттлинг в режиме URLTest → кнопка "Проверить все прокси" (запускает переоценку URLTest). Если заблокирован Telegram API → кнопка "Настройки бота". Если все ОК — кнопки отсутствуют.
- **NEW:** Кулдаун проверки — 2-минутная блокировка между запусками через файл-метку времени `/tmp/podkop_probe_ts`, по аналогии с паттерном `safe_reload`.
- **UX:** Меню диагностики — "Проверить активный исходящий канал" (`Probe Active Outbound`, 🔬) добавлено первым пунктом, над проверкой здоровья апстримов.
- **UX:** Главное меню — имя секции больше не сливается со строкой Активного Маршрута при наличии нескольких секций. Исправлено удаление завершающего переноса строки в heredoc путем использования условного `printf` вместо интерполяции переменной.
- **UX:** Уведомление о запуске — эмодзи часов 🕐 удален из отображения задержки в строке Путь Бота.
- **UX:** Расширена подсказка для ввода пользовательского прокси — теперь показывает все поддерживаемые форматы: `socks5://`, `socks5h://`, `socks5h://hostname:port`, `http://`, `https://`, просто `IP:PORT`. Объясняет, что `socks5h` рекомендуется (удаленный DNS). Отмечает его роль (tier3) в цепочке транспорта.
- **UX:** Обновлена подсказка для ввода фоллбек SOCKS — упоминается поддержка хостнеймов, текст сокращен и реструктурирован для ясности.

## v0.13.93

- **FIXED:** Регрессия в списке исходящих — все прокси показывали "Unknown | N/A", а внутренние узлы групп (`main-urltest-out`) просачивались в список. Причина: конвейер `jq` использовал `.proxies[$sel].all | to_entries[]`, что смещало контекст от корня JSON, заставляя последующие поиски `.proxies[name]` читать из объекта записи вместо корня. Исправлено добавлением `. as $root` в начале `jq`-выражений `total` и `page_tsv`, с последующим использованием `$root.proxies[...]` везде. Затронуло режимы Selector и URLTest; карточки `px_view` не пострадали, так как используют отдельные прямые запросы.
- **FIXED:** `do_px_auto_urltest` (кнопка "Переключить на авто URLTest") искала группу URLTest глобально по всем прокси Clash API, выбирая первую найденную. В многосекционных конфигурациях это могло активировать группу URLTest из другой секции (например, `main` вместо `antiz`). Теперь область поиска ограничена членами `$root.proxies[$sel].all[]` текущей активной секции.
- **FIXED:** Режим единого URL (`proxy_config_type=url`) добавлял новые URL в `proxy_string` вместо замены. В UI было написано "Это заменит любой существующий URL", но код выполнял `printf '%s\n%s' "$existing" "$safe_link"`. Исправлено на `uci set proxy_string="$safe_link"` — строгая семантика единственной ссылки теперь соответствует UI.
- **FIXED:** В подсказке для режима Outbound Config (SSH) был жестко прописан путь `podkop.MAIN.outbound_json=...`. Заменено на `podkop.${sec}.outbound_json=...`, чтобы подсказка отражала фактическое имя активной секции.
- **NEW:** Функция `podkop_dns_check()` — проверяет здоровье туннеля после перезагрузки через fakeip DNS-пробу (`nslookup fakeip.podkop.fyi 127.0.0.42`, ожидает `198.18.x.x`). Позаимствовано из `podkop_autoupdater` (VizzleTF). Теперь используется в `do_reload_podkop`: после перезагрузки показывает "✅ Туннель ОК" или "⚠️ Проверка туннеля не пройдена" вместо просто "Перезагружено!".
- **FIXED:** Сравнение версий в `cmd_check_update` использовало `sort -V`, что не гарантируется в BusyBox. Заменено на покомпонентное числовое сравнение (разбивка `x.y.z`, сравнение каждой части через `-gt`/`-eq`).
- **UX:** Карточка статуса (`cmd_status`) — 🐶 эмодзи для строки Podkop, 📦 для Sing-box, 📨 для строки здоровья Telegram, 🐧 перед названием ОС. Эмодзи часов 🕐 удален из аптайма и задержки маршрута бота. Строки WAN и Public IP объединены в одну (Public IP показывается только если отличается от WAN). Описание здоровья Telegram изменено с "TG direct / via SOCKS / SOCKS" на "direct / tunnel / SOCKS".

## v0.13.28 — P1 Переработка Транспорта (Transport Overhaul)

- **NEW:** `api_request_fast()` — выделенный быстрый путь для команд sendMessage/edit/answerCB/deleteMessage: "липкий" (sticky) таймаут подключения 2с/полный (full) 3с, максимальный таймаут 8с.
- **NEW:** `api_poll_long()` — выделенный путь опроса для `getUpdates`: "липкий" таймаут подключения 3с/полный 4с, максимальный таймаут 65с. Никогда не делит состояние (state) с быстрым путем.
- **NEW:** `LAST_ROUTE_FAST` / `LAST_ROUTE_POLL` / `LAST_ROUTE_DOC` — раздельное отслеживание маршрутов. Функция `api_document()` больше не "отравляет" состояние для быстрого маршрута или маршрута опроса (fast/poll).
- **NEW:** Уровень 2_N (Tier 2_N): UCI список `fallback_socks` (`podkop_bot.settings.fallback_socks`) перебирается по порядку между Podkop SOCKS и `custom_proxy`. Старые уровни были перенумерованы: `tier3=custom`, `tier4=direct`, `tier5=emergency`.
- **NEW:** Счетчик `RECOVERY_MODE` (от 0 до 4): после сбоя всех транспортов (All transports FAILED), следующие 4 цикла опроса будут агрессивно проверять доступность уровней SOCKS, прежде чем пробовать "Прямой" (direct) путь.
- **NEW:** Функция `probe_socks_upstream()` — проверка по 3 эндпоинтам (gstatic x2 + cloudflare). Это снижает количество ложных срабатываний об "упавшем" SOCKS ("SOCKS down"), если заблокирован только один эндпоинт.
- **NEW:** Гистерезис с защитой от "дребезга" (Anti-flap hysteresis) 2/2 для оповещений SOCKS и TG: предупреждение срабатывает только после 2 последовательных проверок с одинаковым состоянием.
- **NEW:** Структурированный файл `SOCKS_STATE_FILE` (параметры `tg/socks/route/last_ok` в формате "ключ=значение"). Экран состояния "Здоровье Туннеля" (Tunnel Health) считывает его и отображает раздельный статус для TG/SOCKS.
- **NEW:** Менеджер фоллбэков SOCKS в Настройках бота (Bot Settings): добавление/просмотр/удаление записей (Интерфейс: Bot Settings -> кнопка Fallback SOCKS).
- **FIXED:** Функция `check_health()` использует прямой (direct) `curl`, а не собственный стек транспорта бота, чтобы не мешать значениям `LAST_ROUTE_FAST/POLL` во время проверок.
- **FIXED:** В меню "Здоровье Туннеля" отдельно отображаются ключи маршрутов (route keys) для путей Poll (опрос) и Fast (быстрый).
- **FIXED:** При уведомлении о запуске (startup notify) используется `api_request_fast`, а быстрый маршрут логируется при успешном подключении.
